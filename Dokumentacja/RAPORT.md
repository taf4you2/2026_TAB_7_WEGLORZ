## 2. Cel projektu

Celem projektu jest zbudowanie kompletnego **systemu bazodanowego wraz z aplikacjami klienckimi**
obsługującego sprzedaż i obsługę biletów oraz karnetów na stacji narciarskiej.

System rozwiązuje następujące problemy:

- **Sprzedaż uprawnień** (biletów jednorazowych i karnetów wielodniowych/sezonowych) w kasie
  oraz online przez przeglądarkę.
- **Obsługa nośników RFID** — wydawanie kart z puli, pobieranie i zwrot kaucji, blokowanie
  i odblokowywanie kart oraz karnetów.
- **Kontrola dostępu na bramkach** wyciągów — weryfikacja ważności nośnika przy odbiciu karty,
  rejestracja przejazdów, mechanizm anti-passback.
- **Raportowanie i statystyki** — raporty zmianowe kasjera, raporty zarządcze administratora,
  historia przejazdów narciarza, statystyki ruchu i przychodów.
- **Zarządzanie infrastrukturą i polityką cenową** — definiowanie wyciągów, tras, harmonogramów
  oraz taryfikatorów przez role zarządcze.

Domeną biznesową jest pełen cykl życia uprawnienia narciarza:
**karta RFID → karnet/bilet → rezerwacja → transakcja → przejazdy przez bramki → ewentualny zwrot kaucji**.

## 3. Zakres wykonanych prac

Zakres prac obejmował zaprojektowanie schematu bazy danych, implementację głównego API,
osobnego API bramki oraz klientów (web i konsolowy klient testowy).

**Warstwa danych**
- Model dziedziny: 27 klas encji EF Core + `SkiResortDbContext` w `SystemStacjiNarciarskiejDLL`.
- Schemat bazy zarządzany skryptami SQL (bez migracji EF): schemat → sekwencje → dane testowe.
- 27 tabel (w tym 9 tabel słownikowych), klucze obce, sekwencje `GENERATED BY DEFAULT AS IDENTITY`.

**SystemAPI (główne API)** — 15 kontrolerów, m.in.:
- Uwierzytelnianie JWT + haszowanie haseł BCrypt (logowanie narciarza, kasjera, administratora).
- Sprzedaż biletów (UC1) i karnetów (UC2), blokada/odblokowanie i zwrot karnetów (UC3, UC11).
- Obsługa puli kart RFID (wydanie, partia, blokada, zwrot kaucji).
- Zakup online dla narciarza, rezerwacje, profil i karty użytkownika.
- Zarządzanie wyciągami, trasami, harmonogramami, taryfami, bramkami.
- Statystyki (dashboard, ruch godzinowy, trend przychodów, kanały sprzedaży, popularność karnetów).
- Raporty: zmianowy kasjera, historia transakcji, historia przejazdów, oczekujące zwroty.

**BramkaAPI (API bramki)** — weryfikacja dostępu, pobranie kart, sprawdzenie karty po ID,
lista aktywnych kart.

**Klienci**
- Portal narciarza (web): `login.html`, `register.html`, `narciarz.html` + interaktywny
  walkthrough (`walkthrough.js`) — system podpowiedzi onboardingowych.
- Mockupy stanowiska kasjera (`mockups/kasjer.html`) oraz panelu administratora.
- `Bramka` — konsolowy klient testowy BramkaAPI.

**Pozostałe**
- Konteneryzacja: `docker-compose` (postgres 17, system-api, bramka-api).
- Dokumentacja: diagram klas, diagram przypadków użycia, scenariusze, opis przypadków użycia.
- Zestaw scenariuszy testów manualnych API (18 scenariuszy, `Scenariusze_Testow_API.md`).

## 4. Analiza problemu, decyzje projektowe i technologie

### 4.1. Analiza problemu i podjęte decyzje

Stacja narciarska wymaga systemu, który łączy **sprzedaż stacjonarną (kasa)** i **online (przeglądarka)**
z **kontrolą dostępu w czasie rzeczywistym na bramkach**. Kluczowe decyzje projektowe:

1. **Rozdzielenie odpowiedzialności na dwa API.** `SystemAPI` obsługuje sprzedaż, raporty i
   zarządzanie, natomiast `BramkaAPI` odpowiada wyłącznie za weryfikację nośnika na bramce.
   Pozwala to skalować i wdrażać kontrolę dostępu niezależnie od logiki sprzedaży oraz traktować
   bramkę jako odrębną, „cienką” usługę przy czytniku.
2. **Współdzielony model dziedziny.** Oba API korzystają z jednej biblioteki encji
   (`SystemStacjiNarciarskiejDLL`) i jednego schematu bazy, co eliminuje duplikację wiedzy
   o danych.
3. **Schemat zarządzany skryptami SQL, nie migracjami EF.** Świadoma decyzja: jednoznaczne,
   wersjonowane skrypty inicjalizujące (`schemat → sekwencje → dane testowe`) ułatwiają
   powtarzalny start środowiska w Dockerze i pełną kontrolę nad DDL.
4. **Tabele słownikowe zamiast enumów w kodzie.** Statusy, typy i wyniki weryfikacji są
   trzymane w 9 tabelach słownikowych, co umożliwia zmianę wartości bez rekompilacji.
5. **Model nośnika rozdzielony od uprawnienia.** `Card` (fizyczny nośnik RFID z puli) jest
   oddzielony od `SkiPass` (uprawnienie), dzięki czemu jedna karta może być w czasie ponownie
   wydawana, a kaucja jest rozliczana niezależnie od ceny karnetu.
6. **Uwierzytelnianie JWT + BCrypt.** Tokeny JWT (bezstanowe, wylogowanie po stronie klienta)
   oraz haszowanie haseł BCrypt dla trzech ról: narciarz, kasjer, administrator.

**Opis.** Model dziedziny obejmuje 27 encji w czterech grupach:

  dodawane przy starcie API).

### 4.4. Uzasadnienie technologii i porównanie alternatyw

**Wybrany stos:** ASP.NET Core (.NET 10) + Entity Framework Core + PostgreSQL 17,
klienci web w czystym HTML/CSS/JavaScript, konteneryzacja Docker Compose.

| Kryterium | ASP.NET Core (wybrany) | Spring Boot (Java) | Django (Python) |
|-----------|------------------------|--------------------|-----------------|
| Język / ekosystem | C# / .NET, statyczne typowanie | Java / JVM, statyczne typowanie | Python, dynamiczne typowanie |
| ORM | EF Core (LINQ, silnie typowany) | Hibernate / Spring Data JPA | Django ORM (zwięzły) |
| Wydajność | Bardzo wysoka (Kestrel) | Wysoka | Średnia |
| Krzywa wejścia | Niska–średnia | Średnia–wysoka (konfiguracja) | Niska |
| Integracja z PostgreSQL | Npgsql — dojrzała | JDBC/Hibernate — dojrzała | psycopg — dojrzała |
| Narzędzia/IDE | Rider, Visual Studio | IntelliJ | PyCharm / VS Code |

**Uzasadnienie wyboru ASP.NET Core:**

- **Spójność i typowanie.** Wspólny język C# dla API i modelu dziedziny oraz silnie typowany
  EF Core ograniczają liczbę błędów na styku z bazą w porównaniu z dynamicznym Django ORM.
- **Wydajność i konteneryzacja.** Lekki serwer Kestrel i gotowe `Dockerfile` ułatwiają wdrożenie
  jako mikrousługi (dwa niezależne API).
- **Mniejszy narzut konfiguracji niż Spring Boot.** Minimal hosting model (`Program.cs`) daje
  zwięzłą konfigurację DI, CORS, JWT i SignalR bez rozbudowanego boilerplate.
- **Dobre wsparcie PostgreSQL** przez Npgsql oraz natywne wsparcie OpenAPI.

Django odrzucono ze względu na dynamiczne typowanie i ścisłe powiązanie z własnym ORM/adminem,
co utrudniałoby rozdział na dwa niezależne API. Spring Boot byłby równoważny technicznie, lecz
wiązałby się z większym narzutem konfiguracyjnym przy tej skali projektu.

### 5.5. Schemat działania aplikacji — scenariusze

Poniższe scenariusze pochodzą z `Dokumentacja/scenariusze/` (zestaw `Scenariusze_Testow_API.md`,
18 scenariuszy `ST-01`…`ST-18`) i zostały dostosowane do roli artefaktu w raporcie. Stanowią one
zarówno opis działania, jak i procedury testów manualnych.

**Scenariusze proste**

- **Logowanie (ST-13).** Kasjer/narciarz wysyła `POST /api/auth/login` z `email`, `password`, `role`.
  Po poprawnej weryfikacji BCrypt API zwraca `200 OK` z tokenem JWT, identyfikatorem i rolą;
  przy błędnych danych `401 Unauthorized`.
- **Sprzedaż biletu jednorazowego (ST-02, UC1).** `POST /api/bilety` z `cardId`, `tariffId`,
  `validOn`, `quantity`. Wynik: `200 OK`, w bazie nowy rekord `reservation` oraz `ski_pass`
  o statusie `aktywny`; dla nieistniejącej karty `400 Bad Request`.
- **Odbicie karty na bramce (UC5/UC6).** Bramka odpytuje BramkaAPI o status nośnika; weryfikacja
  ważności decyduje o przyznaniu lub odmowie dostępu (anti-passback przy próbie ponownego
  przejścia).

**Scenariusze problematyczne**

- **Zwrot karnetu i kaucji (ST-06/ST-07, UC11/UC12).** Najpierw `GET /api/karnety/{id}/symulacja-zwrotu`
  (podgląd kwoty bez skutków ubocznych), następnie `POST /api/karnety/{id}/zwrot`. Logika musi
  poprawnie rozliczyć proporcjonalny zwrot oraz, opcjonalnie, kaucję za zwróconą kartę.
- **Blokada karnetu (ST-05, UC3).** `POST /api/karnety/{id}/blokuj` ze wskazaniem powodu;
  zablokowany karnet musi być odrzucany na bramce.
- **Sprzedaż karnetu end-to-end przez mockup (ST-18, UC2).** Pełny przepływ od wyboru taryfy
  i karty, przez utworzenie rezerwacji, po zapis transakcji.

> Pełna treść 18 scenariuszy: `Dokumentacja/scenariusze/Scenariusze_Testow_API.md`.
