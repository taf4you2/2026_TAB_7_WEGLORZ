Wyobraź sobie, że jesteś narciarzem. Oto Twoja podróż przez ten system:

1

Rezerwujesz online

2

Moduł 2

Trzej aktorzy systemu

Każdy dobry system ma wyraźnie podzielone role. Tutaj mamy trzech "aktorów" — każdy ma jeden, dobrze zdefiniowany obowiązek.

🎭
Metafora: Straż pożarna

Dyspozytor (BramkaAPI) odbiera zgłoszenia. Strażacy (Bramka) są na miejscu zdarzenia. Regulamin (DLL) mówi wszystkim, czym jest "pożar", "wóz" i "hydrant". Każdy robi swoje.

🖥️

Bramka

Aplikacja konsolowa. Widzi klawiaturę, wyświetla zegar, wczytuje ID karty.

🌐

BramkaAPI

Serwer webowy. Przyjmuje zapytania HTTP, pyta bazę danych, odsyła odpowiedź.

📦

SystemDLL

Biblioteka modeli. Nie robi nic sama — ale dostarcza definicje, z których korzystają pozostali.

Aktor #1 — Terminal bramkowy

Plik: Bramka/Program.cs

KOD 

string url = $"http://localhost:49226/api/bramka/sprawdz-karte/{cardId}";

Task<HttpResponseMessage> getTask = client.GetAsync(url);

PO POLSKU

Buduje URL do BramkaAPI, wklejając ID karty do ścieżki.

 

Wysyła zapytanie HTTP GET do API — ale nie czeka bezczynnie.

 

Dopóki odpowiedź nie wróci, odświeża zegar w nagłówku co 200ms...

...żeby ekran nie wyglądał na zamrożony podczas czekania.

 

💡
Wzorzec: programowanie asynchroniczne

Terminal nie "zamraża się" czekając na odpowiedź — robi coś pożytecznego (odświeża zegar). To jak pisanie e-maila podczas czekania na odpowiedź, zamiast siedzenia z wpatrzonymi oczami w skrzynkę.

Aktor #2 — BramkaAPI

Plik: BramkaAPI/Controllers/BramkaController.cs

KOD 

[HttpGet("sprawdz-karte/{id}")]
public async Task<IActionResult> SprawdzKarte(string id)
{
    var karta = await _context.Cards
        .FirstOrDefaultAsync(c => c.Id == id);

    if (karta == null)
        return NotFound(new { Wiadomosc = "Karta nie istnieje." });

    bool czyAktywna = karta.StatusId == 1;
    return Ok(new { Aktywna = czyAktywna });
}
PO POLSKU

Ta funkcja obsługuje zapytania GET pod ścieżką /sprawdz-karte/{id}.

Przyjmuje ID karty jako parametr w adresie URL.

 

Pyta bazę danych: "Czy istnieje karta o tym ID?"

Bierze pierwszą pasującą kartę (lub nic, jeśli nie ma).

 

Jeśli karta nie istnieje — odsyła błąd 404 z wyjaśnieniem.

 

StatusId == 1 oznacza "aktywna". Sprawdza to jak strażnik na liście.

Zwraca odpowiedź: aktywna = tak lub nie.

Aktor #3 — Biblioteka modeli (DLL)

Plik: SystemStacjiNarciarskiejDLL/Models/Card.cs

KOD 

[Table("card")]
public class Card
{
    [Key]
    public string Id { get; set; } = null!;

    public int? StatusId { get; set; }
    public virtual DictCardStatus? Status { get; set; }

    public virtual ICollection<SkiPass> SkiPasses { get; set; }
}
PO POLSKU

Ta klasa odpowiada tabeli "card" w bazie danych.

 

[Key] = "To jest unikalny identyfikator" — jak numer PESEL dla karty.

Id to ciąg znaków (tekst), np. numer seryjny na karcie RFID.

 

StatusId to liczba (1 = aktywna, 2 = nieaktywna itd.).

Status to pełny obiekt statusu — Entity Framework pobiera go sam z bazy.

 

Karta może mieć wiele karnetów (SkiPasses) przez lata użytkowania.

💡
Kluczowe pojęcie: Model danych

Model danych to blueprint dla danych — jak formularz, który musi być wypełniony. Zamiast mówić "mam jakiś obiekt z jakimiś polami", mówisz: "mam Card z Id, StatusId i SkiPasses". Precyzja pomaga AI generować właściwy kod.

Co to jest HTTP i JSON?

Terminal i BramkaAPI rozmawiają przez sieć. Używają do tego dwóch konwencji:

📨

HTTP — język konwersacji

Zapytanie: GET /api/bramka/sprawdz-karte/SKI-123

Odpowiedź: 200 OK (lub 404 Not Found)

📦

JSON — format paczki z danymi

Odpowiedź API wygląda tak:

{ "Aktywna": true, "Wiadomosc": "Karta jest aktywna." }
KOD 

string jsonResponse = await response.Content.ReadAsStringAsync();

using JsonDocument doc = JsonDocument.Parse(jsonResponse);
JsonElement root = doc.RootElement;

if (root.TryGetProperty("aktywna", out JsonElement prop)
    || root.TryGetProperty("Aktywna", out prop))
{
    czyAktywna = prop.GetBoolean();
}
PO POLSKU

Odczytuje treść odpowiedzi jako tekst (ciąg JSON).

 

Zamienia tekst JSON w strukturę, po której można "chodzić" jak po drzewie.

Bierze korzeń tej struktury.

 

Szuka pola o nazwie "aktywna" (małymi literami)...

...albo "Aktywna" (wielką literą). Sprawdza obydwa warianty, bo JSON jest case-sensitive!

 

Jeśli znalazł — odczytuje wartość jako true/false.

Co jeśli coś pójdzie nie tak?

System obsługuje trzy różne scenariusze porażki — każdy inaczej:

!

Karta nie istnieje w bazie

API zwraca HTTP 404. Terminal wyświetla czerwony napis: "ODMÓWIONO DOSTĘPU (Karta nie istnieje)".

!

Karta istnieje, ale jest nieaktywna

API zwraca 200 OK, ale z "Aktywna": false. Terminal wyświetla: "ODMÓWIONO DOSTĘPU".

?

Brak połączenia z API

Terminal łapie HttpRequestException i wyświetla żółty komunikat o błędzie połączenia. Nie crashuje!

💡
Zasada: każdy błąd powinien dać czytelny komunikat

Dobry system nie "wysypuje się" po cichu — pokazuje użytkownikowi, co poszło nie tak i jakim kolorem. Trzy różne scenariusze błędu = trzy różne kolory w terminalu. To wzorzec warty skopiowania w każdym projekcie.

Moduł 4

Baza danych — rdzeń systemu

Wszystkie dane narciarzy, karty, karnety, rezerwacje — żyją w jednym miejscu. Jak to jest zorganizowane i dlaczego to ma znaczenie?

📚
Metafora: Główna księga rachunkowa

Baza danych to jak wielka księga rachunkowa w tradycyjnej firmie — każda transakcja, każda karta, każdy skan jest zapisany w odpowiedniej kolumnie, każdy wiersz to inny rekord. Zmieniasz jedną pozycję — zmiana jest widoczna wszędzie.

Co siedzi w bazie danych?

System ma 20+ tabel. Oto te, które są w centrum akcji:

💳

card

Fizyczna karta RFID. Ma unikalne ID (tekst), status (1=aktywna, 2=nieaktywna) i datę dodania.

🎫

ski_pass

Karnet narciarski. Powiązany z kartą, taryfą i rezerwacją. Ma daty ważności i status.

🚪

gate_scan

Log każdego skanowania bramki. Kto, kiedy, przy której bramce, z jakim wynikiem.

📋

reservation

Rezerwacja zakupu karnetu. Łączy użytkownika z zamówieniem, ma numer i status.

💰

transaction

Każda operacja finansowa: zakup, zwrot depozytu. Powiązana z kasjerem i rezerwacją.

📖

dict_*

Słowniki: statusy kart, typy karnetów, trudność tras itd. Dają nazwy liczbom-statusom.

Entity Framework — tłumacz między C# a bazą danych

API nie pisze ręcznie zapytań SQL. Używa Entity Framework Core — warstwy, która zamienia kod C# na SQL automatycznie.

KOD C# 

var karty = await _context.Cards
    .Select(c => new
    {
        c.Id,
        c.StatusId,
        c.PhysicalCondition,
        c.AddedToPoolAt
    })
    .ToListAsync();
CO TO ROBI

Pyta bazę danych o wszystkie karty w tabeli "card"...

 

...ale nie pobieramy całych obiektów Card — tylko wybrane kolumny.

 

Chcemy: ID, status, stan fizyczny i datę dodania do puli.

 

 

 

ToListAsync = wykonaj zapytanie i wróć z listą wyników (asynchronicznie).

-- Co Entity Framework generuje w tle: SELECT id, status_id, physical_condition, added_to_pool_at FROM card

DbContext — zarządca połączeń

Plik: SystemStacjiNarciarskiejDLL/SkiResortDbContext.cs

KOD 

public class SkiResortDbContext : DbContext
{
    public DbSet<Card> Cards { get; set; }
    public DbSet<SkiPass> SkiPasses { get; set; }
    public DbSet<Gate> Gates { get; set; }
    public DbSet<GateScan> GateScans { get; set; }
    public DbSet<Reservation> Reservations { get; set; }
    public DbSet<Transaction> Transactions { get; set; }
}
PO POLSKU

DbContext to "brama" do bazy danych. Dziedziczy gotowe mechanizmy.

 

Każde DbSet to "okno" do jednej tabeli. Cards = tabela "card".

SkiPasses = tabela "ski_pass". Przez te okna czytasz i piszesz dane.

Gates = tabela "gate". Fizyczne bramki wjazdowe.

GateScans = tabela "gate_scan". Historia skanowań.

Reservations = tabela "reservation". Zamówienia online.

Transactions = tabela "transaction". Historia płatności.

 

💡
DbContext jako "pojedynczy punkt prawdy"

Cały system rozmawia z bazą danych przez jeden DbContext. Dzięki temu masz jeden punkt, który kontroluje dostęp do danych — łatwo logować, łatwo testować, łatwo zmienić bazę danych.

Jak tabele są ze sobą powiązane?

Tabele to nie izolowane wyspy — są ze sobą połączone kluczami obcymi. Oto główna sieć powiązań:

👤
user → reservation Użytkownik może mieć wiele rezerwacji
📋
reservation → ski_pass Rezerwacja rodzi karnet, karnet ma daty ważności
💳
ski_pass → card Karnet jest załadowany na fizyczną kartę RFID
🚪
card → gate_scan Każde skanowanie karty przy bramce jest logowane
🎿
lift ↔ trail Wyciągi i trasy są połączone przez tabelę lift_trail (jeden wyciąg, wiele tras)

Moduł 5

Docker — jak to wszystko uruchomić

System składa się z trzech projektów i bazy danych. Docker sprawia, że wszystko startuje jedną komendą — bez ręcznej konfiguracji środowiska.

📦
Metafora: Kontenery na statku towarowym

Docker to jak system kontenerów morskich: każda usługa (API, baza danych) jedzie w swoim szczelnym kontenerze. Kontener na każdym porcie wygląda tak samo. Zero "u mnie działało!"

🐘

Kontener: postgres

Baza danych PostgreSQL 17. Uruchamia się z gotowym schematem SQL przy pierwszym starcie.

🌐

Kontener: bramka-api

ASP.NET Core Web API. Czeka na PostgreSQL (healthcheck), potem startuje na porcie 49226.

Plik docker-compose.yml — przepis na środowisko

Plik: docker-compose.yml

KOD 

services:
  postgres:
    image: postgres:17
    environment:
      POSTGRES_DB: ski_resort
      POSTGRES_USER: postgres
    volumes:
      - ./Dokumentacja/baza_danych/baza_danych_kod.sql:
          /docker-entrypoint-initdb.d/01_schema.sql
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
PO POLSKU

Definicja usług (kontenerów) do uruchomienia.

Usługa "postgres" — baza danych.

Użyj gotowego obrazu PostgreSQL w wersji 17.

Zmienne środowiskowe: nazwa bazy i użytkownik.

 

 

Skopiuj plik SQL do katalogu startowego — baza sama wykona schemat przy pierwszym starcie!

 

Healthcheck: co 5 sekund sprawdza, czy baza jest gotowa do połączeń.

Jak API startuje po bazie?

DOCKER-COMPOSE 

  bramka-api:
    build:
      context: .
      dockerfile: BramkaAPI/Dockerfile
    ports:
      - "49226:8080"
    environment:
      ConnectionStrings__DefaultConnection:
        "Host=postgres;Database=ski_resort;"
    depends_on:
      postgres:
        condition: service_healthy
PO POLSKU

Usługa "bramka-api" — nasz serwer API.

Zbuduj obraz z pliku Dockerfile...

...znajdującego się w katalogu BramkaAPI.

Port zewnętrzny 49226 → port wewnętrzny 8080.

 

Przekaż string połączeniowy do bazy...

..."postgres" to nazwa usługi w sieci Docker!

 

Nie startuj, dopóki...

...baza danych nie przejdzie healthchecka.

Koniec wyścigu między API a bazą.

💡
Wzorzec: depends_on z condition

Bez depends_on API mogłoby spróbować połączyć się z bazą danych, zanim ta jest gotowa — i wysypać się z błędem. service_healthy czeka na zielone światło z healthchecka. To wzorzec wart zapamiętania przy każdej usłudze zależnej.
