Moduł 2

Trzej aktorzy systemu

Każdy dobry system ma wyraźnie podzielone role. Tutaj mamy trzech "aktorów" — każdy ma jeden, dobrze zdefiniowany obowiązek.

🎭
Metafora: Straż pożarna

Dyspozytor (BramkaAPI) odbiera zgłoszenia. Strażacy (Bramka) są na miejscu zdarzenia. Regulamin (DLL) mówi wszystkim, czym jest "pożar", "wóz" i "hydrant". Każdy robi swoje.

🖥️

Bramka

Aplikacja konsolowa. Widzi klawiaturę, wyświetla zegar, wczytuje ID karty.

🌐

BramkaAPI

Serwer webowy. Przyjmuje zapytania HTTP, pyta bazę danych, odsyła odpowiedź.

📦

SystemDLL

Biblioteka modeli. Nie robi nic sama — ale dostarcza definicje, z których korzystają pozostali.

Aktor #1 — Terminal bramkowy

Plik: Bramka/Program.cs

KOD 

string url = $"http://localhost:49226/api/bramka/sprawdz-karte/{cardId}";

Task<HttpResponseMessage> getTask = client.GetAsync(url);

while (!getTask.IsCompleted)
{
    OdswiezNaglowek();
    await Task.Delay(200);
}
PO POLSKU

Buduje URL do BramkaAPI, wklejając ID karty do ścieżki.

 

Wysyła zapytanie HTTP GET do API — ale nie czeka bezczynnie.

 

Dopóki odpowiedź nie wróci, odświeża zegar w nagłówku co 200ms...

...żeby ekran nie wyglądał na zamrożony podczas czekania.

 

💡
Wzorzec: programowanie asynchroniczne

Terminal nie "zamraża się" czekając na odpowiedź — robi coś pożytecznego (odświeża zegar). To jak pisanie e-maila podczas czekania na odpowiedź, zamiast siedzenia z wpatrzonymi oczami w skrzynkę.

Aktor #2 — BramkaAPI

Plik: BramkaAPI/Controllers/BramkaController.cs

KOD 

[HttpGet("sprawdz-karte/{id}")]
public async Task<IActionResult> SprawdzKarte(string id)
{
    var karta = await _context.Cards
        .FirstOrDefaultAsync(c => c.Id == id);

    if (karta == null)
        return NotFound(new { Wiadomosc = "Karta nie istnieje." });

    bool czyAktywna = karta.StatusId == 1;
    return Ok(new { Aktywna = czyAktywna });
}
PO POLSKU

Ta funkcja obsługuje zapytania GET pod ścieżką /sprawdz-karte/{id}.

Przyjmuje ID karty jako parametr w adresie URL.

 

Pyta bazę danych: "Czy istnieje karta o tym ID?"

Bierze pierwszą pasującą kartę (lub nic, jeśli nie ma).

 

Jeśli karta nie istnieje — odsyła błąd 404 z wyjaśnieniem.

 

StatusId == 1 oznacza "aktywna". Sprawdza to jak strażnik na liście.

Zwraca odpowiedź: aktywna = tak lub nie.

Aktor #3 — Biblioteka modeli (DLL)

Plik: SystemStacjiNarciarskiejDLL/Models/Card.cs

KOD 

[Table("card")]
public class Card
{
    [Key]
    public string Id { get; set; } = null!;

    public int? StatusId { get; set; }
    public virtual DictCardStatus? Status { get; set; }

    public virtual ICollection<SkiPass> SkiPasses { get; set; }
}
PO POLSKU

Ta klasa odpowiada tabeli "card" w bazie danych.

 

[Key] = "To jest unikalny identyfikator" — jak numer PESEL dla karty.

Id to ciąg znaków (tekst), np. numer seryjny na karcie RFID.

 

StatusId to liczba (1 = aktywna, 2 = nieaktywna itd.).

Status to pełny obiekt statusu — Entity Framework pobiera go sam z bazy.

 

Karta może mieć wiele karnetów (SkiPasses) przez lata użytkowania.

💡
Kluczowe pojęcie: Model danych

Model danych to blueprint dla danych — jak formularz, który musi być wypełniony. Zamiast mówić "mam jakiś obiekt z jakimiś polami", mówisz: "mam Card z Id, StatusId i SkiPasses". Precyzja pomaga AI generować właściwy kod.
