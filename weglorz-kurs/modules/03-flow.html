Co to jest HTTP i JSON?

Terminal i BramkaAPI rozmawiają przez sieć. Używają do tego dwóch konwencji:

📨

HTTP — język konwersacji

Zapytanie: GET /api/bramka/sprawdz-karte/SKI-123

Odpowiedź: 200 OK (lub 404 Not Found)

📦

JSON — format paczki z danymi

Odpowiedź API wygląda tak:

{ "Aktywna": true, "Wiadomosc": "Karta jest aktywna." }
KOD 

string jsonResponse = await response.Content.ReadAsStringAsync();

using JsonDocument doc = JsonDocument.Parse(jsonResponse);
JsonElement root = doc.RootElement;

if (root.TryGetProperty("aktywna", out JsonElement prop)
    || root.TryGetProperty("Aktywna", out prop))
{
    czyAktywna = prop.GetBoolean();
}
PO POLSKU

Odczytuje treść odpowiedzi jako tekst (ciąg JSON).

 

Zamienia tekst JSON w strukturę, po której można "chodzić" jak po drzewie.

Bierze korzeń tej struktury.

 

Szuka pola o nazwie "aktywna" (małymi literami)...

...albo "Aktywna" (wielką literą). Sprawdza obydwa warianty, bo JSON jest case-sensitive!

 

Jeśli znalazł — odczytuje wartość jako true/false.

Co jeśli coś pójdzie nie tak?

System obsługuje trzy różne scenariusze porażki — każdy inaczej:

!

Karta nie istnieje w bazie

API zwraca HTTP 404. Terminal wyświetla czerwony napis: "ODMÓWIONO DOSTĘPU (Karta nie istnieje)".

!

Karta istnieje, ale jest nieaktywna

API zwraca 200 OK, ale z "Aktywna": false. Terminal wyświetla: "ODMÓWIONO DOSTĘPU".

?

Brak połączenia z API

Terminal łapie HttpRequestException i wyświetla żółty komunikat o błędzie połączenia. Nie crashuje!

💡
Zasada: każdy błąd powinien dać czytelny komunikat

Dobry system nie "wysypuje się" po cichu — pokazuje użytkownikowi, co poszło nie tak i jakim kolorem. Trzy różne scenariusze błędu = trzy różne kolory w terminalu. To wzorzec warty skopiowania w każdym projekcie.
